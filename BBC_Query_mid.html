Questions

If the operating system processes 32-bit words at a time, then what is the advantage to storing information in an '8-bit' char? doesn't this actually still get stored in a 32-bit word?

If we need to deal with ranges, wouldn't we have to generate compressed output?

How will we fit into her code? Should we do a black-box style, or should we write our own query engine?

We're struggling with dealing with compressed vs uncompressed output. In the case of WAH querying one is able to subtract a fill of a certain length if they are of the same type, and this saves a great amount of time. But with BBC, subtracting from fill lengths in the case of equal fills and incrementing the output fill by this amount doesn't necessarily translate to the output in the same way, and it could just be faster to perform the bitwise operation on every single byte.

Is looking ahead worth it? using 'if' statements could add to time complexity. if we know how long an output fill will be, then it could be less time consuming to perform a single subtraction on two ints than to bitwise OR a sequence of a thousand or more bytes. Then we can just see how long the fill should be, and copy that length of bytes into the output array.

In the case that looking ahead has no conceivable advantage, then it may not even be necessary to initialize such an elaborate ActiveRun structure. We would still need to know the fill len and tail len, but we wouldn't need some of the extra parameters.
